## [Voice AI Systems Vulnerable to Hidden Audio Attacks](https://spectrum.ieee.org/voice-ai-audio-attacks) ⭐️ 8.4/10

Researchers have demonstrated that voice AI systems, including speech-to-text and large language models with audio capabilities, can be hijacked by hidden adversarial audio examples that are imperceptible to humans but cause the AI to transcribe malicious commands. This vulnerability exposes a critical security gap in voice AI applications such as virtual assistants, transcription services, and audio-based LLMs, potentially enabling attackers to remotely control devices or extract sensitive information through seemingly benign audio. The attacks build on prior work on adversarial audio examples, and recent studies have shown the ability to inject malicious audio into live voice chats with AI in real time. Defenses remain challenging due to the limited audibility of perturbations.

hackernews · SVI · May 18, 11:51 · [Discussion](https://news.ycombinator.com/item?id=48178378)

**Background**: Adversarial attacks on AI systems involve crafting inputs that are slightly modified to cause misclassification. In audio, this means adding imperceptible noise to a sound clip that makes a speech-to-text model transcribe a different phrase. This concept is well-known in computer vision, but now applied to voice AI.

## [Using Git's --author flag to block AI bot spam in GitHub repos](https://archestra.ai/blog/only-responsible-ai) ⭐️ 8.2/10

A blog post from Archestra describes how to use Git's --author flag to identify and block AI bot spam in GitHub repositories by checking commit author email addresses against contributor records. This technique offers a practical, low-tech solution for maintainers to filter out automated spam pull requests, which have become a growing problem especially in repos with bounties. However, community discussion highlights that granting contributor status based on email matching can introduce security risks. The method relies on GitHub linking a commit to a user's profile if the commit's author email matches their GitHub account email, thereby granting contributor status and bypassing first-time contributor approval requirements. This approach can be circumvented if a contributor changes their email address or if a malicious actor gets a trivial change accepted.

hackernews · ildari · May 18, 15:24 · [Discussion](https://news.ycombinator.com/item?id=48181125)

**Background**: Git's --author flag allows filtering commits by author when using commands like git log. GitHub uses commit email matching to determine a user's contribution status for permissions such as avoiding approval requirements for fork PR runs. AI bots have been increasingly targeting GitHub repos with automated low-quality pull requests, often for spam or bounty farming.
